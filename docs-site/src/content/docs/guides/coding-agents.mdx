Waired points the coding agents you already use at your own model. There are two
pieces:

- **`waired link`** sets up the per-user integration files — Claude Code skills
  (`~/.claude/skills/`) and the OpenCode provider plugin
  (`~/.config/opencode/plugin/waired.js`) — and issues the gateway token they
  use.
- **Managed settings** point Claude Code at your local model by writing Claude
  Code's system-wide `managed-settings.json`, so you don't set any environment
  variable or shell alias.

## Claude Code

Claude Code is routed through **managed settings**: Waired writes Claude Code's
system-wide `managed-settings.json` so that Claude Code talks to your local
model. No credential is written there, so Claude Code keeps using your existing
claude.ai subscription — including subscription auto-mode (opusplan / Max
Opus→Sonnet usage fallback). There are **no certificates, no `/etc/hosts`
edits, and no shell changes**. This works the same on Linux, macOS, and Windows.

An interactive `waired init` asks whether to enable this **at the end of
install**, once the engine setup and the initial model download have finished
(non-interactive installs enable it with the up-front integration consent);
you can also manage it directly.

<Steps>
1. Set up the integration files and gateway token:

   ```sh
   waired link claude-code
   ```

2. Make sure managed settings are enabled (`waired init` usually did this already):

   <Tabs syncKey="os">
   	<TabItem label="Linux/macOS">
   		```sh
   		sudo waired claude enable
   		```
   	</TabItem>
   	<TabItem label="Windows">
   		```powershell
   		# run elevated:
   		waired claude enable
   		```
   		The installer's “Route Claude Code through Waired” task does this for
   		you by default.
   	</TabItem>
   </Tabs>

3. Just start Claude Code — no `ANTHROPIC_BASE_URL` and no alias needed:

   ```sh
   claude
   ```
</Steps>

<Aside type="caution" title="Waired never breaks Claude">
	If your local serving is paused, disabled, or unavailable, requests **fail
	open** to the real Anthropic API, so Claude Code keeps working. Because no
	credential is set in managed settings, your claude.ai subscription and its
	auto-mode are preserved. You stay in control of when your own model is used.

	If a request can't be served locally (a shape your local model doesn't
	support, or a transient error), Waired retries it against the real Anthropic
	API so the turn still completes. `waired claude status` shows when this last
	happened.
</Aside>

<Aside type="note" title="Which model answers locally">
	Claude Code keeps showing and requesting Anthropic model names (Opus, Sonnet,
	Fable, …). When a request is served locally, Waired answers it with **your
	device's active local model** — whatever `waired status` shows as active —
	regardless of the model the session selected. The Anthropic name only matters
	when the request goes to the real API.
</Aside>

<Aside type="note" title="Your local model's context window">
	Local models usually have a smaller context window than Claude's hosted
	models. Waired tells Claude Code your active model's real window, so Claude
	Code automatically compacts (summarizes) the conversation to stay within it —
	a long session keeps working instead of silently dropping the start of your
	context. If a single turn is still too large to fit, Claude Code compacts and
	retries automatically.
</Aside>

### Switch where requests go, mid-session

You don't have to restart Claude Code to change where your requests are served.
Enabling the integration installs a **`/waired-route` slash command** — run it
inside Claude Code and the *next* request honours the change:

- `/waired-route auto` — the default: prefer Waired, with the automatic
  fallback to the real Anthropic API above.
- `/waired-route waired` — force your Waired inference; never contact Anthropic.
- `/waired-route anthropic` — use the real Anthropic API (handy if the local
  model is misbehaving on a task).

This sets the **main conversation**; subagents follow it unless you set them
separately. The same switch is available from a terminal, where `--subagents`
routes bulk subagent work independently:

```sh
waired claude route auto                          # main conversation
waired claude route anthropic --subagents waired  # main on Anthropic, subagents on Waired
waired claude route                               # show the current policy
```

You can also change it from the tray: the **Claude Code** menu has a route
picker for the **Main conversation** and, separately, **Subagents** (which
default to *Same as main*). The current choice is marked, and a change takes
effect from the next request — no restart.

<Aside type="tip" title="Keep requests strictly on Waired">
	Prefer that a request never silently goes to Anthropic when it can't be
	served locally? Choose the `waired` route — it uses your Waired inference
	only and shows you the local error instead of redirecting:

	```sh
	waired claude route waired               # main conversation, never Anthropic
	waired claude route --subagents waired   # subagents only
	```
</Aside>

<Aside type="note" title="Which Waired machine?">
	`waired` means your Waired inference — *which* machine (this device or a mesh
	peer) is chosen by `waired worker`, not here.
</Aside>

<Aside type="tip" title="Shaping a machine from the web console">
	In the [web console](https://app.waired.ai/), open a machine's page to shape
	how the mesh uses it for Claude Code:

	- **Serve main / sub-agents** — two toggles decide whether the mesh may route
	  the main conversation and/or sub-agent work to this machine. Turn main off to
	  keep a slower machine for parallel sub-agent grunt work only. (This only
	  affects requests routed *from other machines*; the machine still serves its
	  own local Claude Code use.)
	- **Max concurrent requests** — how many requests the machine serves at once.
	  Raising it lets the machine help more peers in parallel, but each parallel
	  request needs its own VRAM and slows every request, so the page shows a
	  recommended maximum and warns before you exceed it.
</Aside>

### See where requests go, in Claude Code

So you never have to wonder whether Waired is active, enabling the integration
also adds two things you see **inside Claude Code**:

- A **status line segment** in the footer showing the current route — e.g.
  `⚡ waired: on Waired (qwen3-8b-instruct)` (served on your Waired network, with
  the model that answered the last request), `→ waired: Anthropic` (using
  the real API), or `✕ waired: agent down`. If you don't already have a custom
  status line, Waired adds this for you; if you do, `waired init` and
  `waired claude enable` ask before touching it (or run
  `waired claude statusline install --wrap` yourself).
  It disappears cleanly if Waired is later removed.

  One caveat: Claude Code uses only the **highest-precedence** status line, and a
  project's `.claude/settings.local.json` / `.claude/settings.json` outranks the
  user-level one Waired installs. Waired never edits project files — run
  `waired claude status` inside the project and it reports the shadowing along
  with a one-liner you can paste into your own status-line script to show the
  route there.
- A one-line **notice after any turn that fell back** to the real Anthropic API,
  so a request that couldn't be served locally is never silently redirected.

Both are removed by `waired claude disable`. To skip the status line at enable
time, pass `--no-statusline`.

## OpenCode

OpenCode is configured by the same `waired link` step, which installs the
provider plugin at `~/.config/opencode/plugin/waired.js`. The plugin points
OpenCode at your Local Gateway and forwards the gateway token for you.

```sh
waired link opencode
```

Run `waired link` with no agent name to set up every installed agent at once.

## Verify and remove

Check that routing is active:

```sh
waired claude status
```

It reports whether managed settings are enabled and where the
`managed-settings.json` file lives:

- **Linux:** `/etc/claude-code/managed-settings.json`
- **macOS:** `/Library/Application Support/ClaudeCode/managed-settings.json`
- **Windows:** `C:\Program Files\ClaudeCode\managed-settings.json`

The tray's **Claude integration** menu shows the same status.

To remove the integration (surgical — it only undoes what `link` added):

```sh
waired unlink            # remove all agents
waired unlink claude-code  # remove just one
```

To stop routing Claude Code through Waired (reverts the managed-settings change,
restoring Claude Code's default behavior):

```sh
sudo waired claude disable   # Windows: run elevated, no sudo
```

If a coding agent isn't using your model, run [`waired doctor`](/troubleshooting/) —
it checks the gateway token, paused state, engine readiness, mesh peer, and the
integration in one pass, and can repair what's fixable.
