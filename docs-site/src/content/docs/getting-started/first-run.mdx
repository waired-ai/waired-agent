**Installing normally does this for you.** The install command starts Waired
and then goes straight into sign-in, so a first-time setup is covered by the
[Quickstart](/quickstart/) and you never type `waired init` yourself.

This page is for the times you do:

- setup was interrupted and you are resuming it
- you installed with `--no-init`
- you are setting up a server with no desktop
- you want to know what each choice actually does

<Aside type="tip" title="Same account = same network">
	Sign in to **every** computer with the **same Google account**. That shared
	sign-in is what lets one machine use another's AI. Different accounts make
	separate, unconnected networks.
</Aside>

## Start it

<Tabs syncKey="os">
	<TabItem label="Windows">
		Open **Terminal (Administrator)** — right-click the Start button — and
		run:

		```powershell
		waired init
		```

		<Aside type="note" title="“waired is not recognized”">
			The installer added Waired to your system path, but a terminal
			opened *before* the install still has the old one. Open a **new**
			Administrator terminal. If you would rather not, run it by full
			path: `& "C:\Program Files\Waired\waired.exe" init` — adjusting the
			folder if you chose a different install location.
		</Aside>
	</TabItem>
	<TabItem label="macOS">
		```sh
		sudo waired init
		```
	</TabItem>
	<TabItem label="Linux">
		```sh
		sudo waired init
		```
	</TabItem>
</Tabs>

It prints a sign-in link and waits:

<Expected>
```
Sign in using this link:
  https://app.waired.ai/login/…

>> Press Enter to open your browser (or open the link above yourself)...
```
</Expected>

Press **Enter** and choose your Google account. You can also copy the link into
any browser, on this computer or another one.

<Aside type="caution" title="Leave that window open until setup finishes">
	Installing the AI software needs administrator rights, and a web page cannot
	grant them. **This command is what actually performs those steps** while the
	browser page drives them.

	If you close it, the browser says *“The setup command on … was closed before
	this finished. Your progress was saved.”* Run the same command again to carry
	on — nothing is lost, and any model download continues regardless.
</Aside>

## Finish in the browser

After sign-in your browser lands on the setup page for this computer. Pick a
model, press **Yes, set it up**, and it reports what the computer is doing,
live:

<Screenshot
	src="setup-wizard-progress.png"
	alt="The Waired setup page in a browser, showing three steps — Install the AI software (done), Download the AI model (in progress, with a bar reading 7.9 GB of 19.2 GB), and Check the speed (waiting)."
	caption="Three steps, reported by the computer itself. Closing this tab does not stop them."
/>

| Step | What is happening | How long |
|---|---|---|
| **Install the AI software** | The program that loads and runs models. This is the part that needs administrator rights — which is why the terminal window has to stay open. | A minute or two |
| **Download the AI model** | The AI itself, several gigabytes. | 5–30 minutes |
| **Check the speed** | A short measurement, so Waired knows whether this machine keeps up. | Up to 5 minutes |

<Expected>
	Every step ticked, and **“Everything's ready — this computer can now run AI
	on its own.”** Press **Go to device**.
</Expected>

While the browser is driving, the terminal stays quiet and asks nothing. It
prints one line telling you that you can connect your coding tools later:

```
You can set up your coding tools later from this terminal with `waired link all`.
```

<Aside type="note" title="The setup page is in English">
	The web console is English-only for now. Translations are planned.
</Aside>

## Set up in the terminal instead

At *“Setup is continuing in your browser…”*, press **Enter**. The terminal
takes over.

What it asks depends on how it is running.

### After a normal install

The terminal asks about your coding tools, and nothing else — the model was
chosen in the browser, or by the installer:

<Expected title="You are asked">
	Whether to route Claude Code through Waired. Answer **Y**.

	Answering no leaves the connection installed but inactive; turn it on later
	with `sudo waired claude enable`.
</Expected>

### <a id="no-desktop-no-browser"></a>On a server, or with no browser

With `--no-browser`, `--non-interactive`, or on a machine where Waired cannot
reach a running service to hand over to, `waired init` does the whole job
itself and asks the full set of questions:

<Steps>

1. **Google sign-in.** Only the link and a pairing code are printed — open them
	on another device.

2. **Run AI models on this computer? `[Y/n]`** — **Y** on a desktop or GPU box.
	**N** on a machine that should only *use* another computer's AI; it still
	joins your network. The default follows this machine's memory.

3. **Let your other devices use this computer's AI? `[Y/n]`** — **Y** on a
	machine meant to serve your laptop. This is independent of the previous
	answer.

4. **Let Waired install and manage its own Ollama? `[Y/n]`** — only asked when
	you already have your own Ollama installed. **n** keeps yours and Waired
	uses it as it is.

5. **Route Claude Code through Waired? `[Y/n]`**

6. **Run a quick performance benchmark now? `[Y/n]`** — asked once the model is
	ready. Worth doing: if this computer cannot keep up, Waired offers a lighter
	model and names both with their quality tiers so you can weigh the
	trade-off.

</Steps>

The model then downloads in the foreground with a progress bar:

```
Starting the inference engine…
Preparing to download qwen3-coder-30b-a3b-instruct…
Downloading qwen3-coder-30b-a3b-instruct: 41%  7.9 GB / 19.2 GB (44 MB/s)
```

<Expected>
```

│  Account   you@example.com                                       │
│  Device    my-desktop  d-4f2a…                                   │

```
</Expected>

<Aside type="caution" title="“This machine can only run a very small model”">
	On a low-memory computer, setup may report that only a very small model fits
	and suggest **not** running one. Take that advice: at that size a coding
	model produces broken output more often than useful output. The computer is
	still worth setting up — it can use the AI on your other machines.
</Aside>

<details>
<summary>Options for servers and scripts</summary>

| Flag | Effect |
|---|---|
| `--no-browser` | Never open a browser; print the URL and pairing code |
| `--non-interactive` | Accept the hardware-derived defaults, ask nothing |
| `--no-wait-model` | Return to the shell immediately; download in the background |
| `--inference-bundled-model-id <id>` | Pin a model instead of letting Waired choose |
| `--inference-enabled=true\|false` | Answer the first question without being asked |
| `--share-with-mesh=true\|false` | Answer the second question without being asked |
| `--control <url>` | Point at a non-default service |
| `--mask-pii` | Mask home directory, username, hostname and email in the output, for pasting into a bug report |

</details>

## Signing in from the Waired app instead

The [Waired app](/guides/waired-app/) has a **Log in…** entry, and it does sign
this computer in.

<Aside type="caution" title="It cannot finish a first-time setup on its own">
	Installing the AI software needs administrator rights that the app cannot
	grant. On a computer that has never been set up, signing in from the app
	gets as far as the setup page and then asks you to run `waired init` in a
	terminal anyway.

	Use **Log in…** for signing back in after a sign-out, or on a computer that
	is already set up. For a first setup, use the command above.
</Aside>

## Setting up again

Running setup again on a computer that is already signed in is safe: it
re-authenticates and reopens the setup page, and does **not** count as a new
device against your account.
